# NebulaOps v23.3 — Frontend Runtime Overlay Fix

## Scope

This patch addresses the case where local MFE dist artifacts are valid and Docker images rebuild successfully, but `http://nebulaops.localhost/remotes/<slug>/remoteEntry.js` still serves stale bundles without the NebulaOps auth bridge.

## Changes

- Added `scripts/wsl/sync-v23.3-frontend-runtime.sh`.
- Updated `scripts/wsl/repair-v23.3-frontend-remotes.sh` to sync validated local shell and MFE dist artifacts directly into the running `frontend` container after image rebuild and container recreation.
- Added container/local SHA verification for every MFE `remoteEntry.js`.
- Added runtime sync before same-origin endpoint verification.
- Preserved live-only behavior: no mock/static operational data was added.

## Runtime Behavior

The frontend repair now has three layers of protection:

1. Validate local dist artifacts.
2. Rebuild frontend/MFE nginx images.
3. Copy the validated local dist artifacts into the live frontend container and verify SHA/content before accepting the runtime.

This removes dependency on stale Docker image layers or stale container filesystem state.

## Recovery Command

For an already-running stack that fails only at `remoteEntry.js is missing the NebulaOps auth bridge`, run:

```bash
./scripts/wsl/sync-v23.3-frontend-runtime.sh
```

Then verify:

```bash
./scripts/wsl/start.sh --skip-preflight
```
