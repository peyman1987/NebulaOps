# NebulaOps v23.4 Safe Action Plan Upgrade

## Scope

This upgrade implements Priority 3 for NebulaOps v23.4 while keeping the public product version at `23.4 / 23.4.0`.

The goal is to prevent destructive or disruptive runtime operations from being executed directly from the UI. Docker, Kubernetes and Helm actions now pass through a Safe Action Plan gate before execution.

## New backend endpoints

The gateway now exposes:

- `POST /api/runtime/docker/actions/plan`
- `POST /api/kubernetes/actions/plan`
- `POST /api/platform/actions/execute`

The plan endpoints collect live evidence from Docker Engine, kubectl and Helm before returning an execution plan. The execute endpoint requires the exact confirmation phrase produced by the plan endpoint.

## Safe Action Plan content

Each plan returns the following technical fields:

- `whatWillBeTouched`
- `impactedResources`
- `dependencies`
- `reversible`
- `reversibility`
- `command`
- `apiCall`
- `risk`
- `confirmationPhrase`
- `executionEndpoint`
- `executePayload`
- `planId`
- `realDataOnly`

Plans are generated from live runtime state only. If Docker, kubectl or Helm is unavailable, the plan response reports the unavailable state instead of returning synthetic impact rows.

## Protected Docker actions

The following Docker operations are gated by Safe Action Plan:

- container stop
- container restart
- container kill
- container remove
- Docker project stop
- Docker project restart
- image prune
- volume prune
- network prune
- system prune

The Docker plan includes Docker Engine API details, live impacted container rows, project labels, mounts where available, and prune preview candidates where applicable.

## Protected Kubernetes and Helm actions

The following Kubernetes and Helm operations are gated by Safe Action Plan:

- pod delete
- pod restart
- workload rollout restart
- workload scale
- job delete
- node drain
- node cordon
- Helm rollback
- Helm uninstall
- YAML apply
- YAML delete

The Kubernetes plan includes kubectl authorization checks, live resource lookups, server-side dry-run output for YAML operations, namespace context, related events where applicable, and Helm history/status where applicable.

## Operational Issues integration

Operational issue row actions now include planned actions where the row represents a disruptive operation candidate. Examples include:

- Docker issue row → Plan restart, Plan kill, Plan remove
- Docker project issue row → Plan restart, Plan stop
- Kubernetes pod issue row → Plan restart, Plan delete
- Kubernetes workload issue row → Plan rollout restart, Plan scale
- Helm release issue row → Plan rollback, Plan uninstall

Read-only troubleshooting actions remain direct GET calls. Disruptive actions include `requiresPlan: true` and route to the planning endpoints first.

## Frontend behavior

The generated live-only remote template now handles Safe Action Plan flows consistently:

1. Call the relevant plan endpoint.
2. Render the returned plan in the detail panel.
3. Ask the operator to type the exact confirmation phrase.
4. Call `/api/platform/actions/execute` only after exact confirmation.
5. Show the live execution result and refresh the active view.

The UI does not execute destructive actions from static button metadata alone.

## Version handling

The public release remains `23.4 / 23.4.0`.

The generated remote runtime cache key was moved to:

- `v23.4.3-safe-action-plan-live-real-data-control-plane`

This is only a cache-busting identifier and does not change the public product version.
