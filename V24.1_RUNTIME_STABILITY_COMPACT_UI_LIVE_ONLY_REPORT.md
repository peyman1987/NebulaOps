# NebulaOps v24.1 Runtime Stability, Compact UI and Live-Only Report

## Release objective

NebulaOps v24.1 is a stabilization and operational quality release. The release keeps the existing platform scope and focuses on startup reliability, deterministic version alignment, compact operator visibility, same-origin MFE runtime delivery, and live-only data behavior.

## Implemented changes

### Runtime and release alignment

- Updated visible and runtime release identifiers to `v24.1` / `24.1.0`.
- Updated Docker image names and local runtime names from `nebulaops-v23-2-*` to `nebulaops-v24-1-*`.
- Updated WSL scripts, frontend package metadata, remote verification tooling, generated frontend assets and release validation checks.
- Renamed versioned scripts and reports to the v24.1 namespace.

### Extensions control plane

- `/api/extensions` remains a fast installed-extension registry endpoint and does not perform Kubernetes, Docker or port-forward probes.
- `/api/extensions/summary` now returns cached status when available and otherwise returns non-blocking registry metadata.
- Deep runtime checks are isolated behind explicit operator actions through `/api/extensions/{slug}/status?refresh=true` and `/api/extensions/{slug}/diagnostics`.
- Extension UI status refresh now explicitly requests a live probe only when the operator chooses the Status action.
- APIForge, KubeBridge and Contract Hub remain disabled by default and are started only by explicit operator action.

### Frontend and MFE runtime

- Frontend shell package updated to `nebulaops-v24-1-shell-host@24.1.0`.
- Angular production build script now uses `--progress=false` for deterministic non-interactive builds.
- The compact operator density rules are preserved so the UI provides the visibility benefit of a lower browser zoom without requiring the browser zoom to be changed.
- Sidebar search, command palette, APP BAR and EXTENSIONS split remain active.
- Same-origin classic remote bundles remain verified for all 22 micro frontends.

### Live-only policy

- No frontend mock rows, public demo endpoints, personal placeholder users or bundled operational snapshots were introduced.
- Missing integrations continue to be represented with explicit runtime states such as `NOT_PROBED`, `DISABLED_BY_DEFAULT`, `KUBERNETES_UNAVAILABLE`, `UNAVAILABLE` or `DEGRADED`.
- APIForge runtime data files remain empty by default; operator-created/imported data is mounted at runtime.

## Environment limitation

Backend Maven compilation was not executable in this sandbox because Maven is not installed, Docker is not reachable, and the sandbox cannot install Maven from Debian repositories. The project keeps the existing WSL backend build path through `scripts/wsl/build-backend-jars.sh`, which uses host Maven when available or the Docker-based Maven path when Docker Desktop is running.

Expected backend verification command in the target WSL/Docker Desktop environment:

```bash
./scripts/wsl/build-backend-jars.sh --force
```

The script failed only because the sandbox lacks both host Maven and Docker access, not because of a project validation error.
