# NebulaOps v23.2 — OpenLens Multi-Cluster Kubeconfig Upgrade

## Objective
OpenLens Kubernetes no longer depends only on the single `kubectl` current context mounted in the gateway container. The UI now supports a MongoDB-backed kubeconfig registry so operators can add other Kubernetes clusters and operate each saved configuration from its own tab.

## Implemented capabilities

### Backend
- Added MongoDB persistence in `gateway-service` for kubeconfig records.
- New collection: `kubernetes_kubeconfigs`.
- New API endpoints:
  - `GET /api/kubernetes/kubeconfigs`
  - `POST /api/kubernetes/kubeconfigs`
  - `DELETE /api/kubernetes/kubeconfigs/{id}`
  - `POST /api/kubernetes/kubeconfigs/{id}/probe`
  - `GET /api/kubernetes/helm/releases?namespace=all&clusterId={id}`
- Existing Kubernetes read and action endpoints now support `clusterId`:
  - pods, deployments, services, ingress, configmaps, statefulsets, nodes, namespaces;
  - logs, describe, YAML, restart, scale, cordon, uncordon, drain;
  - apply/delete/diff YAML.
- For saved kubeconfigs, the gateway materializes the selected config into a temporary file for the duration of the command and removes it immediately after execution.
- If MongoDB, kubeconfig, cluster API, certificate, or permissions are unavailable, the response returns an explicit runtime error instead of fake data.

### Frontend OpenLens MFE
- Added **Kubernetes cluster registry** panel.
- Added one tab for the local `kubectl` current context.
- Added one tab per kubeconfig saved in MongoDB.
- Added save form for kubeconfig YAML.
- Added probe/delete actions for saved configs.
- All OpenLens actions now target the selected cluster tab by appending `clusterId` to live API requests.
- No kubeconfig data is stored or duplicated in the frontend bundle.

### Compose/runtime
- Added `KUBECONFIG_MONGO_URI` to gateway service configuration.
- Added MongoDB as a gateway dependency for the kubeconfig registry.

## Why OpenLens previously showed only the master/local cluster
The previous implementation used the gateway container's mounted `KUBECONFIG=/kube/config`, therefore `kubectl` always resolved the current local context. There was no persisted registry for additional kubeconfig files and no `clusterId` parameter for targeting a selected cluster.
