# NebulaOps v23.4 — Extension Async Control Plane Upgrade

## Scope

This upgrade keeps the public release version at **23.4 / 23.4.0** and changes the extension lifecycle model from blocking, synchronous start/restart calls to an asynchronous operation control plane.

The goal is to prevent gateway/UI timeouts during long-running extension deployment workflows and to make each lifecycle phase visible to the operator.

## Runtime behavior

`POST /api/extensions/{slug}/start` now returns immediately with **202 Accepted** and an `operationId`.

The deployment continues in the gateway worker pool and can be observed through:

- `GET /api/extensions/operations/{operationId}`
- `GET /api/extensions/{slug}/events`
- `GET /api/extensions/{slug}/diagnostics`

The same asynchronous model is also used by `POST /api/extensions/{slug}/restart`.

## Operation phases

The operation payload exposes both machine-readable and UI-readable phase fields:

- `STARTING` / `Starting`
- `PULLING_IMAGE` / `Pulling image`
- `APPLYING_KUBERNETES_MANIFESTS` / `Applying Kubernetes manifests`
- `WAITING_FOR_ROLLOUT` / `Waiting for rollout`
- `PROBING_ENDPOINT` / `Probing endpoint`
- `READY` / `Ready`
- `FAILED` / `Failed`
- `TIMEOUT` / `Timeout`

## API contract

### Start extension

```http
POST /api/extensions/{slug}/start
```

Response:

```json
{
  "accepted": true,
  "httpStatus": 202,
  "operationId": "...",
  "extension": "apiforge",
  "action": "start",
  "state": "RUNNING",
  "phase": "STARTING",
  "phaseLabel": "Starting",
  "operationUrl": "/api/extensions/operations/{operationId}",
  "eventsUrl": "/api/extensions/apiforge/events",
  "diagnosticsUrl": "/api/extensions/apiforge/diagnostics",
  "realDataOnly": true
}
```

### Poll operation

```http
GET /api/extensions/operations/{operationId}
```

Returns the current phase, current state, recent events, step outputs and final runtime status when available.

### Extension events

```http
GET /api/extensions/{slug}/events
```

Returns recent operation events for the selected extension. Events are generated by the live lifecycle runner and are not static records.

### Extension diagnostics

```http
GET /api/extensions/{slug}/diagnostics
```

Now includes:

- Docker availability check
- kubectl availability check
- deployment details
- service details
- pod details
- live extension status
- recent async events
- recent async operations

## UI changes

The extension control panel now treats start/restart as asynchronous operations.

When an operator clicks **Start** or **Restart**, the UI:

1. Receives the `operationId`.
2. Displays the async operation card.
3. Polls `/api/extensions/operations/{operationId}`.
4. Shows the phase progression.
5. Refreshes live extension status when the operation reaches `READY`, `FAILED` or `TIMEOUT`.

The UI no longer waits for the original HTTP request to finish the whole Docker/Kubernetes deployment flow.

## Live-only guarantee

No mock extension records or static lifecycle results were introduced.

The registry endpoint remains lightweight by design. Deep status, event history and diagnostics are loaded only when explicitly requested by the operator.

## Smoke verification

Added:

```bash
./scripts/wsl/smoke-extension-async-control-plane-v23.4.sh
```

By default the smoke script verifies non-disruptive endpoints:

- `/api/extensions`
- `/api/extensions/{slug}/events`
- `/api/extensions/{slug}/diagnostics`

The real start flow is optional because it deploys a real extension:

```bash
NEBULAOPS_RUN_ASYNC_EXTENSION_START_SMOKE=1 ./scripts/wsl/smoke-extension-async-control-plane-v23.4.sh
```
