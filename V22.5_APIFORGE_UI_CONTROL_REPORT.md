# NebulaOps v22.5 — APIForge UI-Controlled Extension Startup

## Scope

NebulaOps v22.5 now keeps only one extension under `extensions/`: APIForge.
APIForge can be started and stopped from the NebulaOps UI after the core platform is already running.

## Runtime model

- `./scripts/wsl/start.sh --rebuild` starts the NebulaOps core platform.
- APIForge is not auto-deployed by default.
- APP BAR contains `APIForge Control`.
- `APIForge Control` opens `/api/extensions/console`.
- The console calls gateway endpoints to build, push, apply and rollout APIForge on Kubernetes.

## Gateway endpoints

```text
GET  /api/extensions
GET  /api/extensions/apiforge/status
POST /api/extensions/apiforge/start
POST /api/extensions/apiforge/stop
GET  /api/extensions/console
ANY  /apiforge/**
ANY  /extensions/apiforge/**
```

## Implementation details

The gateway control plane uses the real local runtime:

```text
Docker daemon via /var/run/docker.sock
Kubernetes via /kube/config
kubectl and docker from /opt/nebula-tools
APIForge sources from /workspace/extensions/apiforge
local registry at localhost:5001
```

No mock data, static records, or seeded demo records are introduced.
If APIForge is not deployed, the UI shows `STOPPED` / `NOT_FOUND` states from real Kubernetes commands.

## Startup behavior

Default:

```bash
./scripts/wsl/start.sh --rebuild
```

Starts NebulaOps core and leaves APIForge controlled by UI.

Optional legacy auto-start:

```bash
./scripts/wsl/start.sh --rebuild --with-extensions-k8s
```

## Open APIForge

After clicking `Start APIForge` in the control console:

```text
http://nebulaops.localhost/apiforge/
```

The browser goes through the gateway proxy, so it no longer depends on direct Windows access to `localhost:31110`.
