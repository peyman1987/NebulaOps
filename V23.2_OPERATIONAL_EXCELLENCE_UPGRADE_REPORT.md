# NebulaOps v23.2 — Operational Excellence Upgrade Report

## Scope

This package extends v23.2 after the Incident Command Center and Platform Catalog work with five high-value operational sections:

1. Runtime Readiness Dashboard
2. Docker Storage & Cleanup Center
3. Environment & Configuration Center
4. Dependency & Impact Graph
5. Test & Quality Dashboard

The implementation follows the project rule: no mock or static frontend data. The new remotes load live data from gateway endpoints. When a runtime source is missing, the response contains explicit states such as `UNREACHABLE`, `DOCKER_SOCKET_NOT_FOUND`, `BUNDLE_NOT_BUILT_OR_NOT_MOUNTED`, `REPORT_NOT_GENERATED`, or `NOT_CONFIGURED`.

## Docker Storage & Cleanup behavior

The storage center uses Docker Engine data:

- `/system/df` through the Docker socket;
- containers from `/containers/json?all=true`;
- images from `/images/json`;
- volumes from `/volumes`;
- networks from `/networks`.

Prune preview is derived from live Docker data and identifies stopped containers, dangling images, unused/dangling volumes and custom networks. Volume cleanup remains guarded because volumes can contain real database/cache/broker data.

## Environment & Configuration behavior

Configuration values come from `System.getenv()`. Secret-like values are masked. Missing required settings become explicit risk rows instead of silent fallbacks.

## Dependency & Impact behavior

The dependency graph combines live readiness probes with platform topology edges. If a dependency is down, affected modules are marked degraded with impact text.

## Quality Dashboard behavior

The quality section reads real project files and generated report files under `reports/`. If a report has not been generated, the UI shows `REPORT_NOT_GENERATED` instead of inventing a successful result.
