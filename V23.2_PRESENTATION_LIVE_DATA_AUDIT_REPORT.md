# NebulaOps v23.2 - Presentation and Live Data Audit

## Scope
This audit verifies the v23.2 package after the Platform Catalog and Incident Command Center additions. The objective is to remove mock/static runtime data, improve presentation consistency, and validate the build/package health.

## Live-data corrections applied

### Task Management
- Removed startup seed/demo work items from `WorkItemService`.
- Removed frontend hardcoded `default-org` and `portfolio` values from task creation and board filters.
- Task APIs now accept optional organization/project values and resolve the organization from configuration when omitted.

### Observability and Incident Command Center
- Removed `default-org` frontend/runtime assumptions.
- Incident and observability views keep empty sources empty instead of generating synthetic rows.
- Correlation, task, audit, notification and Kubernetes links remain driven by real endpoints.

### DevSecOps
- Removed static SBOM components.
- SBOM generation now calls Trivy with CycloneDX output when available.
- Image scan no longer defaults to a preselected image. If no image is supplied, the API reports a live-only tool status instead of scanning a fake/default image.
- Reports endpoint no longer returns a fake ready report when no real report source exists.

### AI Ops
- Replaced misleading `fallback` response field with `localAnalysis` for local heuristic analysis when no external LLM provider is configured.

### Identity/Auth
- Local admin shortcut is disabled by default and must be explicitly enabled with `nebulaops.auth.local-admin.enabled=true`.
- Removed the static `peyman@nebulaops.local` frontend user placeholder.
- Auth health reports v23.2.

### Platform Catalog and Presentation
- Remote manifest ports were aligned with the runtime platform catalog registry.
- `platform-catalog`, `infra-hub`, `release-center`, `progressive-delivery`, `policy-center`, `notification-center`, `identity-admin` and `incident-command-center` now have non-conflicting metadata ports.
- The UI link wording was adjusted from implementation-specific `remoteEntry.js` to `Runtime bundle`.

## Static/mock scan result
A repository scan was executed for terms such as `mock`, `dummy`, `fake`, `seed`, `fixture`, `default-org` and `portfolio` across frontend, backend, scripts and configuration, excluding generated `dist`, `node_modules`, package locks and generated remote bundles.

Remaining matches are intentionally non-runtime categories:
- Mockito-based unit tests.
- Comments documenting that no seed/mock data is produced.
- Validation scripts that detect mock/static patterns.
- UI messages stating that no mock/fallback data is served.
- Configurable default organization property using `nebulaops`, not the old `default-org` value.

No remaining match was identified as a runtime dataset or frontend-owned fake data source.
