# NebulaOps v23.4 — Improved Extensions Integration Report

## Scope

The existing `extensions/` directory was replaced with the improved extension package supplied by the operator:

- APIForge
- KubeBridge
- Contract Hub
- `extensions/extensions.manifest.json`

The integration keeps the v23.4 operating rule: no mock/static runtime data and explicit runtime states when an extension is not running or not reachable.

## Default state

All installed extensions are disabled by default.

Implementation details:

- every extension entry in `extensions/extensions.manifest.json` contains `enabledByDefault: false` and `defaultState: DISABLED`;
- every extension-local Kubernetes manifest starts with `replicas: 0`;
- `infrastructure/kubernetes/apiforge.yaml` also starts with `replicas: 0`;
- `scripts/wsl/start.sh` keeps `NEBULAOPS_EXTENSIONS_K8S=false` by default;
- `ExtensionControlController` exposes `enabledByDefault=false` and `defaultState=DISABLED` in extension status responses;
- Start/Restart explicitly applies the manifest and scales the selected deployment to `replicas=1`;
- `scripts/wsl/deploy-extensions-k8s.sh` explicitly scales selected extensions to `replicas=1` only when the deployment script is intentionally executed.

## Live-only corrections

APIForge persistent package files remain empty at startup:

- `extensions/apiforge/data/collections.json`
- `extensions/apiforge/data/environments.json`
- `extensions/apiforge/data/history.json`

The improved APIForge `StorageService` no longer seeds default environments such as Development/Staging/Production or any external demo URL. Operators create or import real environments from the UI/API.

## Integration updates

- The extension control panel still controls APIForge, KubeBridge and Contract Hub from the separate EXTENSIONS button.
- Extensions remain outside the main APP BAR service cards.
- Kubernetes manifests remain local to each extension under `extensions/<slug>/k8s/deployment.yml`.
- Gateway extension Start/Restart flow now supports disabled-by-default manifests by scaling to one replica before rollout status.
- Validation scripts now fail when extension manifests do not use `replicas: 0` by default.
- Package validation now fails when an extension is not marked disabled by default in the manifest.
