# NebulaOps v23.2 — MFE Runtime Auto-Repair Hotfix

## Problem observed

During `./scripts/wsl/start.sh`, the platform started successfully but the live MFE runtime verification reported many remotes as:

```text
missing, stale, or still serving ESM syntax
```

The backend build was successful. The issue was isolated to stale frontend/MFE runtime Docker images or stale same-origin static `remoteEntry.js` bundles served by the frontend Nginx container.

## Root cause

The v23.2 shell serves MFE bundles from the frontend image under:

```text
/remotes/<slug>/remoteEntry.js
```

When local sources were updated but the frontend runtime image was not rebuilt without cache, the browser could still receive old ESM Angular output or missing bundles. The previous package also referenced `repair-v23.2-frontend-remotes.sh`, but the only available repair implementation was the the maintained `repair-v23.2-frontend-remotes.sh`, and that script did not cover all 22 v23.2 remotes.

## Fixes applied

- Added `scripts/wsl/repair-v23.2-frontend-remotes.sh`.
- The repair script now covers all 22 v23.2 MFE services.
- The repair script rebuilds local frontend/MFE dist artifacts.
- The repair script removes stale frontend/MFE containers.
- The repair script removes stale frontend/MFE Docker images.
- The repair script rebuilds frontend/MFE images with `--no-cache`.
- The repair script recreates frontend/MFE containers.
- The repair script verifies every served `/remotes/<slug>/remoteEntry.js` endpoint.
- `scripts/wsl/repair-v23.2-frontend-remotes.sh` is now a compatibility wrapper to the maintained v23.2 repair script.
- `scripts/wsl/start.sh` now auto-runs the v23.2 MFE repair when stale/missing/ESM bundles are detected.
- `scripts/wsl/start.sh` now fails fast if a repair still leaves invalid bundles, instead of allowing blank MFE pages.
- `scripts/wsl/build-frontend-images.sh` now includes all 22 v23.2 MFE services.

## Operational behavior

By default, `start.sh` now performs automatic MFE repair if it detects invalid served bundles.

To disable automatic repair intentionally:

```bash
NEBULAOPS_AUTO_REPAIR_MFE=false ./scripts/wsl/start.sh
```

Manual repair command:

```bash
./scripts/wsl/repair-v23.2-frontend-remotes.sh
```

If Chrome still shows old content after repair, clear site data for `nebulaops.localhost` or hard-refresh with `Ctrl+Shift+R`.
