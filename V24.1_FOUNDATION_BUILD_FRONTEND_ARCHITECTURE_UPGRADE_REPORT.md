# NebulaOps v24.1 — Foundation, Build Performance and Frontend Architecture Upgrade

## Scope

This package promotes NebulaOps from the previous local-runtime baseline to the public release identity:

```text
v24.1 / 24.1.0 / nebulaops-v24-1
```

The upgrade covers three areas:

1. Foundation release alignment.
2. Build performance improvements for the frontend and runtime images.
3. Frontend architecture standardization through shared contracts.

## Foundation upgrade

The release identity is aligned across project documentation, frontend package metadata, remote manifests, Maven coordinates, Docker Compose project names, image names, WSL script names, runtime assets and validation reports.

Primary guards:

```bash
python3 scripts/validate-version-alignment-v24.1.py
./scripts/wsl/smoke-version-alignment-v24.1.sh
```

The validator checks stale release identifiers, frontend package/lockfile agreement, Maven versions, Compose project/image names, runtime assets and manifest versions.

## Build performance upgrade

The frontend now uses a root-managed dependency installation and selective rebuild cache:

```bash
./scripts/wsl/build-frontend-changed.sh
./scripts/wsl/build-frontend-local.sh
```

Implementation details:

- one root frontend `package-lock.json` is the canonical dependency lock;
- remote Angular builds reuse the root `frontend/node_modules` through PATH/NODE_PATH;
- `.build/frontend-build-cache-v24.1.json` tracks shell and remote input hashes;
- unchanged MFE dist folders are skipped;
- `NEBULAOPS_FORCE_FRONTEND_DIST_BUILD=true` forces a full shell/MFE rebuild;
- npm cache is centralized under `.build/npm-cache` unless `NEBULAOPS_NPM_CACHE` is provided.

Startup modes:

```bash
./scripts/wsl/start.sh --core
./scripts/wsl/start.sh --full
./scripts/wsl/start.sh --rebuild
./scripts/wsl/start.sh --core --rebuild
```

`start.sh --rebuild` remains full by default to avoid a behavior regression. `--core` starts only identity, data stores, messaging, gateway and frontend. Extended backend services, observability tools and standalone MFE containers are enabled by the full profile set.

Docker Compose profiles added in `docker-compose.yml` and `infrastructure/docker-compose.yml`:

- `full` — extended backend/runtime services;
- `observability` — Prometheus, Loki, Tempo, OpenTelemetry Collector and Grafana;
- `tools` — native local tool UIs;
- `mfe` — standalone MFE containers;
- existing optional profiles remain `gitlab`, `sso-proxy` and `sso-prometheus`.

## Frontend architecture upgrade

Shared frontend contracts are now present under `frontend/libs`:

| Library | Purpose |
| --- | --- |
| `nebulaops-api-client` | Central API client with bearer token injection, timeout, correlation id and live-envelope enforcement. |
| `nebulaops-live-state` | Standard source state model: `READY`, `DEGRADED`, `UNAVAILABLE`, `NOT_CONFIGURED`, `ERROR`, `LOADING`. |
| `nebulaops-mfe-runtime` | Shared MFE bootstrap helper and live-only payload guard. |
| `nebulaops-ui-kit` | UI Kit v2 tokens, action bar contract, side panel contract, compact density CSS and source state classes. |

The frontend architecture guard is included in preflight:

```bash
node frontend/tools/verify-frontend-architecture-v24.1.mjs
```

## Live-only rule

No mock or static operational data was added. The v24.1 architecture continues to require runtime data from gateway/service endpoints, Docker, Kubernetes or configured tools. Missing integrations must appear as explicit `UNAVAILABLE`, `NOT_CONFIGURED` or `DEGRADED` states.
